# 操作手册（OPERATIONS）

这份手册讲**日常怎么运维**、**怎么看懂 edge 报告**、以及**上真钱前必须逐条勾掉的检查清单**。

---

## 一、日常运维

### 启动挂机（积累数据 + 只提示）

```powershell
cd D:\winsey_code\pmCrypto
python main.py
```

- 让它一直跑。每轮日志写入 `logs/pmcrypto.log`，数据写入 `data/pmcrypto.db`。
- 网页面板：http://127.0.0.1:8000

### 看它在干什么

- **终端 / 日志**：`logs/pmcrypto.log` 每轮一条"抓取 N 个市场 / 有无信号"。
- **网页面板**：运行状态、建议信号、候选机会、最新市场快照。
- **看信号**：manual 模式下，凡是"建议下单"都是**提示**，需要你人工判断，系统不会自动下单。

### 常见现象与判读

| 现象 | 含义 | 要不要担心 |
|---|---|---|
| "本轮无可提示信号" | 没有满足条件的机会 | **正常**，机会本就稀少 |
| "本轮跳过 N 个市场" | 部分市场无有效盘口 | 少量正常；大量需查 API |
| "未取得任何有效市场" | API 连通/字段映射有问题 | **要查**（见下） |
| 网页面板打不开（端口占用） | 上个进程没释放 8000 | 不影响主循环；重启或换端口 |
| 系统进入"急停" | 触发了熔断（API失败/亏损/连亏） | **要查**急停原因再决定重启 |

### 端口被占用怎么办

若日志出现 `Errno 10048 ... 8000`，是上一个进程没释放端口。主循环不受影响。要恢复面板：
- 关掉旧的 python 进程，或
- 改 `config.py` 里 `web_port` 换一个端口。

### 解除急停

急停后系统只监控、不产生信号。排查完原因后，重启进程即可（`STATE` 是进程内状态，重启即重置）。**不要在没查清急停原因前盲目重启。**

## 二、edge 验证（整个项目最重要的一步）

方向性策略（阶段 B）**上真钱前，必须先用你自己积累的真实数据验证它到底有没有 edge**。步骤：

### 第 1 步：积累足够数据

挂机跑一段时间（越久越好，尤其要覆盖一些**市场收盘**的周期，才有结算结果）。判断数据量：

```powershell
python -c "import sys; [x.reconfigure(encoding='utf-8') for x in (sys.stdout,) if hasattr(x,'reconfigure')]; sys.path.insert(0,'.'); from data.store import Store; s=Store('data/pmcrypto.db'); print('市场数:', len(s.distinct_market_ids()), '| 结算数:', len(s.all_resolutions()))"
```

### 第 2 步：拉取结算结果

```powershell
python main.py --refresh-resolutions
```

对已收盘的市场，记录谁赢了（收盘后价格收敛到 ≈1 的结果为获胜方）。**没有结算结果就无法验证 edge**，所以这一步要在市场有收盘后反复跑。

### 第 3 步：跑 edge 报告

```powershell
python main.py --edge-report
```

### 怎么读这份报告

```
下注 N 笔 | 胜率 X% | 平均单笔收益 ±0.xxxx | 标准差 0.xxxx
平均收益 95% 置信区间 [下界, 上界]
判定：edge 显著为正 [OK]  /  证据不足 [NO]
```

**判定规则（系统自动执行，不靠你感觉）：**
- **[OK] edge 显著为正** ＝ 下注数 ≥ 30 **且** 置信区间下界 > 0。
- **[NO] 证据不足** ＝ 样本太少（<30）**或**置信区间跨越 0。

**关键提醒：**
- 胜率高、平均收益为正，**但样本 < 30 → 依然是 [NO]**。别被小样本的漂亮数字骗了。8 笔全赢在统计上毫无意义。
- 置信区间下界 > 0 才说明"扣掉运气成分后，edge 大概率真实存在"。
- **只有 [OK] 才允许进入下一道门槛（小额灰度）。[NO] 就继续积累数据 / 回炉调信号，绝不上真钱。**

## 三、上真钱前的检查清单（切 auto 前逐条勾掉）

> 这是不可跳过的关卡。任何一条没勾掉，都不要把 `executor_mode` 改成 `auto`。

### 门槛 ① — edge 验证

- [ ] `--edge-report` 判定为 **[OK] edge 显著为正**
- [ ] 下注样本数 ≥ 30（越多越可信）
- [ ] 置信区间下界明显 > 0（不是勉强擦过 0）
- [ ] 结算数据覆盖了足够多不同的市场（不是一两个市场的偶然）

### 门槛 ② — 小额灰度（真跑但极小仓位）

- [ ] `auto.py` 已实现真实下单逻辑（当前是拒绝执行的空壳）
- [ ] 用**远小于正常**的仓位真跑一段时间
- [ ] 对比"实际成交价 vs 系统估算价"，校准滑点模型
- [ ] 实盘表现与回测预期**方向一致**（不能回测赚、实盘亏）

### 密钥与安全

- [ ] `.env` 里配置的是**专用小额钱包**私钥，**不是主资产钱包**
- [ ] 确认 `.env` 在 `.gitignore` 内、没进过 git、没出现在日志
- [ ] 钱包里只放你**输光也不心疼**的金额

### 风控就位

- [ ] 熔断线（`halt_balance_daily/total`）设成你能接受的水平
- [ ] `kelly_max_single_pct` 单笔封顶符合你的风险承受
- [ ] `max_consecutive_losses` 连亏熔断已启用

### 全部勾完后

1. 把 `config.py` 的 `executor_mode` 改为 `auto`。
2. **从最保守参数起步**：`kelly_fraction_min/max` 先用低值，先跑小仓位。
3. 紧盯前几十笔，确认行为符合预期，再逐步放开。

## 四、故障排查速查

| 症状 | 可能原因 | 处理 |
|---|---|---|
| `ModuleNotFoundError` | 依赖没装全 | `pip install -r requirements.txt` |
| "未取得任何有效市场" | API 变更/字段映射失效 | 检查 `data/client.py`，可能 SDK 又更新了 |
| 大量市场被跳过 | 盘口拉取失败/限频 | 看跳过样本原因；调大 `poll_interval_sec` |
| 系统频繁急停 | API 失败累积 | 检查网络；看 `core/state` 失败阈值 |
| 中文日志乱码 | 控制台非 UTF-8 | 已内置强制 UTF-8；若仍乱码检查终端设置 |
| edge 报告永远 [NO] | 数据不够/信号无 edge | 继续积累；若长期无 edge，说明信号需重做 |

---

## 五、一句话总结

**先让系统诚实回答"有没有 edge"，再谈下注。** 这套系统最大的价值，是在你没有优势时拦住你别赌，在你有优势时用数学最优方式放大它。耐心走完 edge 验证，比急着上真钱重要一百倍。
