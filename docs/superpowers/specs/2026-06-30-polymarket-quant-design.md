# Polymarket 量化交易系统 — 设计文档（第一阶段定稿）

- 日期：2026-06-30
- 目标：构建一套能持续运行的 Polymarket 量化分析 / 自动交易系统。
- 总路径：**C（自动分析 + 人工确认）→ 回测 → 小额实盘灰度 → A（全自动）**，每一跳都有准入闸门，不允许"裸奔上真钱"。

---

## 0. 核心原则与现实约束（必须始终清醒）

- 本系统的价值：把赌博变成**有纪律、可验证、有刹车**的自动交易，提高"不爆仓、长期为正"的概率。
- **没有任何系统能保证"今晚必涨"。** "必须增长否则拔电源"这一原始约束在工程上无法兑现，已放弃；任何为兑现它而做的高风险操作都被结构性禁止。
- 第一阶段套利策略（Yes/No 互补、互斥概率和偏差）大概率**很少出单**——这是健康信号，说明成本算对了。它的使命是搭"能安全验证任意策略"的地基，而非赚大钱。
- 真正的赚钱主力预期在**阶段 B（方向性 / 价差策略）**，但必须经过**回测 + 小额灰度**两道闸门才能动用真钱。

---

## 1. 阶段划分

| 阶段 | 内容 | 准入门槛 |
|---|---|---|
| **C（第一阶段）** | 自动抓数据 + 偏差检测 + 净收益计算 + 风控过滤 + 终端/日志/网页展示；**只输出建议，不真发单** | 无（只读市场数据，零资金风险） |
| 回测 | 用积累的历史快照验证策略方向性 | C 跑通、数据积累足够 |
| 小额灰度 | 极小仓位真实下单，校准滑点、对比回测预期 vs 实际 | 回测结论可接受 |
| **A（全自动）** | 真·自动下单，全局急停生效 | **回测 + 小额灰度双闸都通过** |

## 3. 系统总览

常驻 Python 程序，主循环每 N 秒做四件事：**抓数据 → 找偏差 → 算净收益 → 出信号**；同时跑一个轻量本地网页面板（只读）。第一阶段下单步骤只输出建议、不真发单。

```
                  ┌─────────────────────────────────────────┐
                  │            常驻主循环 (每 N 秒)             │
  Polymarket ───▶ │  抓数据 ──▶ 找偏差 ──▶ 算净收益 ──▶ 出信号  │
   CLOB API       │   │                              │       │
                  │   ▼                              ▼       │
                  │  SQLite 存储              终端打印 + 日志  │
                  └─────────┬────────────────────────┬───────┘
                            ▼                        ▼
                     本地网页面板(只读) ◀───  信号/持仓/机会(JSON)
```

每轮循环开头检查 `core/state.py` 全局状态：若处于急停态则跳过交易逻辑、仅维持监控。

**关键设计点：**

- **执行层可插拔**：`main.py` 只认 `Executor` 接口。C 注入 `manual`，A 注入 `auto`，切换只改一行配置 `EXECUTOR_MODE`，主循环与策略零改动。
- **风控独立成关**：任何信号变成"建议下单"前必须过 `risk/guard.py`，且 guard 受 `core/state` 急停管辖。
- **策略与数据解耦**：`detector` 只接收标准化 `Market` 对象，不关心数据来源。实时喂实时数据，回测喂历史数据，同一套逻辑——这是回测能低成本接入的基础。
- **加新策略 = 加一个 detector 文件**，主干不动。

## 5. 策略逻辑（detector + pricing）

### 偏差检测（detector.py）

**① Yes/No 互补偏差**
- 理论上 `Yes_ask + No_ask ≈ 1.0`。若明显 `< 1`，同时买 Yes + No 可锁定无风险差价。
- 偏差 = `1 − (Yes_ask + No_ask)`，越大越好。

**② 互斥多结果概率和偏差**
- 多个互斥结果的 `ask` 之和理论 `≈ 1.0`。和明显 `< 1` 时全买可锁利。

### 净收益计算（pricing.py）—— 必须把成本扣干净

```
净收益 = 名义偏差
        − 交易手续费（Polymarket 费率，可配置）
        − 滑点（按盘口深度估算，强制保守：宁可少算机会，不可高估收益）
        − 最小流动性过滤（深度太薄直接丢弃）
若 净收益 ≤ MIN_PROFIT_THRESHOLD → 丢弃，不提示
```

滑点模型预留校准接口，小额灰度阶段用"实际成交价 vs 估算价"反向校准。

---

## 6. 风控与全局急停

### 风控闸门（risk/guard.py）

| 参数 | 作用 | 第一阶段建议值（可配置） |
|---|---|---|
| `MAX_POSITION_PCT` | 单笔最多占账户比例 | 10%（100 USDC → 单笔 ≤ 10） |
| `MAX_TOTAL_EXPOSURE_PCT` | 同时在场总仓位上限 | 50% |
| `MIN_PROFIT_THRESHOLD` | 净收益低于此不提示 | 例如 1.5% |
| `MIN_LIQUIDITY` | 盘口深度低于此直接丢弃 | 可配置 |
| `DAILY_MAX_LOSS_PCT` | 单日亏损触及即熔断停机 | 20%（切 A 后生效） |

第一阶段 C 不真发单，风控主要做"过滤 + 建议仓位"；切 A 后立即变真刹车，无需重写。

### 全局状态 + 急停（core/state.py）

- 维护全局运行状态：正常 / 急停。
- 触发急停的条件：连续亏损超限、API 失败计数超阈值、数据断流、账户余额异动（切 A 后）。
- 主循环每轮检查；`guard` 与未来 `auto` 执行器都受其管辖。

---

## 7. 数据与 API 稳健性

- `client.py` 内置：**限频 + 退避重试 + 失败计数**；失败累积到阈值触发 `core/state` 全局急停。
- 轮询间隔保守起步（30–60s 可配置），避免撞 Polymarket 速率限制导致临时封禁。

---

## 8. 安全（密钥）

- **第一阶段 C 完全不需要私钥**——只读市场数据、不发单，零资金风险，现阶段无需提供任何敏感信息。
- 切 A 时才需要：① 专用 Polygon 钱包私钥（**绝不能是主资产钱包**，单独建只放小额）；② Polymarket API 凭证。
- 密钥仅写入本地 `.env`：**绝不进 git、绝不外传、绝不出现在日志**。`.gitignore` 预先挡死 `.env`、`*.db`。

## 9. 审计发现的关键缺口与对策（写明以备切 A 时不踩）

- **缺口 A — 回测的幸存者/前视陷阱**：快照离散抽样，两次快照间的瞬时机会看不到，会高/低估可成交性；冷启动无历史数据。
  - 对策：回测结论仅作"方向性参考"，**不作必然盈利证明**；切 A 前必须再过**小额实盘灰度**。回测 + 灰度双闸，缺一不可。
- **缺口 B — 滑点是估的，估错就亏**：估算偏乐观则净收益虚高。
  - 对策：滑点估算**强制保守**；灰度阶段用实际成交价反向校准。
- **缺口 C — API 限频/封禁**：常驻轮询撞速率限制可致瘫痪。
  - 对策：`client.py` 限频 + 退避 + 失败熔断，挂到 `core/state` 急停；轮询间隔保守。

# 阶段 B：方向性 edge + 自适应 Kelly 复利（追加设计）

## B.0 核心认知（不可动摇）

- 阶段 A 套利"不预测未来、几乎不亏也几乎不赚"；阶段 B"主动判断误定价、按 Kelly 复利下注、允许亏损但绝不爆仓"。
- **复利是放大器不是发动机**：有 edge 则放大盈利，无 edge 则加速亏损。整个阶段 B 的价值浓缩成一件事——**诚实且尽快地回答 edge 是否存在**。
- "稳步增长"违背零和市场的数学；能承诺的只有"若真有 edge，分数 Kelly 是让长期增长率最大化且不破产的最优策略"（Kelly 1956）。
- **保命是复利成立的物理前提**：一旦归零，(1+r)^n 直接终结。故熔断可放宽、绝不可删。

## B.1 edge 定义

edge = 估计真实概率 p 与市场报价 q 的系统性正偏差。买入成本 q，单笔期望 ≈ p − q。仅当 p > q 有正 edge。

## B.2 统计信号（Polymarket 自身数据，可回测）

| 信号 | edge 假设 | 死法 |
|---|---|---|
| ① 价格动量 momentum | 信息逐步扩散，趋势未走完 | 假突破 |
| ② 深度-价格背离 volume_divergence（成交量代理）| 放量未动=有人建仓 | 洗盘 |
| ③ 买卖盘失衡 book_imbalance | 深度失衡=短期压力方向 | 挂单欺骗 |

合成：`p = clamp(q + Σ wᵢ·confidenceᵢ·deltaᵢ, 0, 1)`。权重初期先验、后由 edge 回测校准。

## B.3 自适应 Kelly（大胆修正①）

- Kelly 比例：`f* = (p − q) / (1 − q)`。
- **自适应分数**：Kelly 分数随合成置信度在 [¼, ½] 间线性放大——侵略性精确跟随证据强度。
- 三重削减：`f_final = min(f*·分数, 单笔20%封顶, 剩余总仓位)`；下注额 = **当前实时余额** × f_final（复利核心）。

## B.4 精选机会（大胆加码②）

- edge 低于 `edge_min_threshold` 直接不出手；每轮只提示 edge 最大的前 N 个。
- 下注质量 > 下注频率，避免被手续费/方差磨死。

## B.5 保命熔断

- 单笔封顶 20% | 余额跌破 60 当日停 | 跌破 30 彻底停 | 连亏 5 笔熔断。全部挂在 `core/state` 全局急停。

## B.6 edge 验证（科学命门，不可跳过）

- `data/resolutions.py`：市场收盘后价格收敛，取价格≈1 的结果为获胜方，写入 resolutions 表。
- `backtest/edge_report.py`：回放历史快照，用真实结算结果结算每笔方向性下注，统计胜率/累计盈亏/**平均收益 95% 置信区间**。
- **判据**：下注数 ≥ 30 且 CI 下界 > 0 → edge 显著为正 → 方可进入小额灰度。否则判"证据不足，不上真钱"。
- **数据积累期不可跳过**：冷启动无结算数据，必须先常驻挂机积累"快照+结算结果"。

## B.8 上线路径（每跳都有闸门）

```
① 常驻挂机积累数据（快照+结算）→ ② 实现阶段B模块 → ③ edge验证闸门（edge_report判正）
→ ④ 小额灰度闸门（真跑几笔校准滑点）→ ⑤ 放开分数Kelly复利（auto执行器实现）
→ ⑥ 叠加"加密现价"外部信号（第二个detector）
```

`execution/auto.py` 在 edge 未验证前继续拒绝执行，作为"防误上真钱"的物理保险。

## B.10 加密现价外部信号（第⑥步，已实现）

最有效的单一外部信号：加密类市场的概率误定价。

- `data/crypto_price.py`：拉 BTC/ETH 等实时现价（Coinbase 主/Binance 备，免费无 key），带缓存与失败降级；年化波动率用先验常数。
- `strategy/signals/crypto_mispricing.py`：解析问题文本 `(asset, strike, direction)` + 市场 `end_ts`，用对数正态（GBM 零漂移）算 `P(S_T>K)=Φ(d2)`，得真实概率 p_true；置信度随临近到期升高。
- `strategy/edge_detector.py::detect_crypto_edge`：对加密市场用 p_true vs 报价 q 找误定价，产出 EDGE_DIRECTIONAL 机会。
- `Market` 增加 `end_ts` 字段（client 从 SDK `state.end_date` 抽取，store 幂等加列）。
- 开关：`config.enable_crypto_signal`。回测 `edge_report` 不传 crypto_source，保持回测纯净（不引入历史现价的前视问题）。

**真实验证结论**：对临近到期的 BTC 市场，模型 p_true 与市场报价高度吻合（误差普遍 <0.05），发现 edge=0——证明模型正确且市场高效，系统诚实地"无机会不下注"。
